## 1. Core Concept

**Vision**: You create a seed file (pyproject.toml, devenv.nix, a config file). Remora detects that the file matches a "template" definition. The template specifies:

1. **Inward expansion**: The seed file's sections become agent-managed nodes (e.g., each `[tool.*]` table in pyproject gets its own node that can reason about and modify its section)
2. **Outward scaffolding**: The seed file triggers creation of downstream files/directories (e.g., pyproject.toml triggers `src/<pkg>/`, `tests/`, `.github/workflows/`)
3. **Cascading**: Generated files may themselves match templates, triggering further generation (a generated `devenv.nix` triggers its own template rules)

The generation logic lives in declarative config (what to trigger) + Grail scripts (how to generate). Execution is configurable: auto-execute for safe scaffolding, propose-and-review for code generation.

## 2. Integration Model Analysis

Three ways to integrate with Remora's existing architecture. You asked for examples, pros/cons, and implications of each.

### Option A: Extend Existing Node Types + Reconciler

**How it works**: Add new `LanguagePlugin` implementations for each template-triggering format (TOML sections, Nix blocks, YAML sections). These plugins produce `CSTNode`s from the seed files just like the Python/Markdown plugins do today. The existing reconciler picks them up, projects them into `Node`s, provisions bundles, and the actor system processes them normally.

**Example**: A `pyproject.toml` with `[tool.ruff]` would:
- `TomlPlugin` already exists and discovers `table` nodes
- The existing tree-sitter query for TOML (`toml.scm`) captures each top-level table
- Each table becomes a CSTNode → Node → Actor
- The `bundle_overlays` config maps `table` nodes to a `pyproject-agent` bundle
- That bundle's tools include Grail scripts for scaffolding

**What changes**:
- Better tree-sitter queries for TOML/Nix/YAML to capture more granular sections
- New bundle overlay rules (already supported via `bundle_rules` config)
- New Grail tool scripts for generation workflows
- Possibly new query files (`.scm`) for finer-grained capture

**Pros**:
- **Minimal code changes** — most machinery already exists
- **Consistent model** — template nodes are just nodes, same lifecycle, same subscriptions, same actor model
- **Bundle overlay rules already support this** — `bundle_rules` with `name_pattern` can route `[tool.ruff]` to a different bundle than `[project]`
- **Cascading is free** — generated files get reconciled automatically, triggering their own templates
- **Testing is straightforward** — same test patterns as existing code

**Cons**:
- **Limited by tree-sitter granularity** — tree-sitter queries can only capture what the grammar exposes. TOML tables are fine, but nested structures or cross-section relationships may be hard to express
- **No explicit "template" concept** — the system doesn't know that `pyproject.toml` is a "template file" vs a regular discovered file. Everything is a node. You lose the ability to say "this is a generator" vs "this is generated"
- **Reconciler doesn't know about generation order** — if pyproject.toml generates `src/app.py`, and `src/app.py` gets reconciled before the generation completes, you get a race
- **Can't express "this node should generate THAT file"** — the current model is reactive (events trigger agents), not declarative (this template produces these outputs)

**Implications**:
- Template behavior is entirely determined by bundle assignment and tool scripts
- No way to track "this file was generated by that template" (no provenance)
- Works great for the "inward expansion" case (sections → nodes), weaker for "outward scaffolding" (generating new files)

---

### Option B: New "Generator" Node Type

**How it works**: Introduce a new `NodeType.GENERATOR` (or `NodeType.TEMPLATE`) that sits above regular code nodes. Generator nodes have special semantics: they declare their outputs, they run before regular nodes, and they can create/manage child nodes programmatically. Think of them as "meta-nodes" that produce other nodes.

**Example**: A `pyproject.toml` generator node would:
- Be discovered by the reconciler as a `GENERATOR` type (matched by filename pattern in config)
- Have a bundle with a `generate.pym` tool that defines the generation workflow
- On trigger, the generator reads its source (the pyproject.toml), runs the generation script, and emits `NodeDiscoveredEvent`s or writes files
- Generated nodes have `parent_id` pointing back to the generator
- The generator can re-run when the seed file changes, reconciling its outputs

**What changes**:
- New `NodeType.GENERATOR` in `types.py`
- New template registry in config: `templates` section mapping file patterns → generator behavior
- Modified reconciler to handle generators before regular nodes
- Generator-specific execution logic (run generation before/after regular reconciliation)
- Provenance tracking (generated_by field on nodes)

**Pros**:
- **Explicit intent** — the system knows "this is a generator" and can treat it differently
- **Provenance** — generated nodes can track their origin ("created by pyproject.toml generator")
- **Ordering guarantees** — generators run first, then their outputs are reconciled
- **Lifecycle management** — when a generator is deleted, its generated artifacts can be cleaned up
- **Composable** — generators can depend on other generators (devenv.nix generator depends on pyproject.toml generator)
- **Re-generation** — when the seed file changes, the generator can diff its previous outputs and update only what changed

**Cons**:
- **Significant new code** — new node type, new reconciler logic, new execution path
- **Two-phase reconciliation** — generators must run before regular discovery, which complicates the reconciler
- **Complexity budget** — adds a new concept that every developer must understand
- **Risk of over-engineering** — might end up building a build system (Bazel, Nix) inside Remora
- **Edge cases** — what happens when a generator's output conflicts with manually-edited files?

**Implications**:
- The node graph becomes two-layered: generators → generated nodes
- Need to decide if generated files are "owned" by the generator (regenerated on change) or "seeded" (generated once, then human-owned)
- Opens the door to DAG-based generation pipelines (generator A → file → generator B → more files)
- Closest to how Nix flakes or Bazel BUILD files work

---

### Option C: Separate Pre-Reconciliation Phase

**How it works**: Add a new phase to the startup/runtime lifecycle that runs *before* the existing reconciler. This "template expansion" phase scans for seed files matching template patterns, runs their generation workflows, writes output files to disk, and then hands off to the normal reconciler to discover everything.

**Example**: A `pyproject.toml` would:
- Template expansion phase scans project root, finds `pyproject.toml` matches pattern `pyproject.toml`
- Runs the associated Grail generation script which reads the file and creates `src/<pkg>/__init__.py`, `tests/conftest.py`, etc.
- Template expansion completes
- Normal reconciler runs, discovers all files (including generated ones) as regular nodes
- Agents process nodes normally

**What changes**:
- New `TemplateExpander` component that runs before `FileReconciler`
- Template registry config (file patterns → generation scripts)
- Generation script runner (Grail-based)
- File locking/ownership tracking (which files are template-managed)

**Pros**:
- **Clean separation** — template expansion is a distinct phase, doesn't complicate the existing reconciler
- **Simple mental model** — "templates run first, then everything else works as before"
- **No new node types** — generated files are just regular files that get regular nodes
- **Easy to debug** — you can run template expansion standalone and inspect the output
- **Familiar pattern** — this is how code generators (protobuf, OpenAPI codegen) work

**Cons**:
- **No runtime reactivity** — templates only expand at startup or on explicit re-run, not reactively when the seed file changes
- **No agent management of sections** — the "inward expansion" case (sections → managed nodes) doesn't fit this model well
- **Two systems to understand** — template expansion + reconciliation are separate pipelines
- **Loses Remora's strengths** — the whole point of Remora is reactive, event-driven agent processing. A batch pre-phase doesn't use any of that
- **Generated file conflicts** — if someone edits a generated file, the next template expansion might overwrite it

**Implications**:
- Works well for initial project scaffolding but poorly for ongoing management
- Essentially a code generator bolted onto Remora, not integrated with it
- The "self-generating" aspect is limited to startup time
- Better suited as a separate CLI tool (`remora scaffold`) than an integrated system

### Comparison Matrix

| Criterion | A: Extend Existing | B: Generator Nodes | C: Pre-Phase |
|-----------|--------------------|--------------------|-------------|
| Code change size | Small | Large | Medium |
| Inward expansion (sections → nodes) | ✅ Natural | ✅ Explicit | ❌ Weak |
| Outward scaffolding (seed → files) | ⚠️ Via tools | ✅ Native | ✅ Native |
| Cascading (generated files → more generation) | ✅ Free | ✅ With ordering | ⚠️ Multi-pass |
| Provenance tracking | ❌ None | ✅ Built-in | ⚠️ External |
| Reactive to changes | ✅ Always | ✅ Always | ❌ Batch only |
| Mental model complexity | Low | Medium-High | Medium |
| Risk of over-engineering | Low | High | Low |
| Fits Remora's architecture | ✅ Perfectly | ⚠️ Extends it | ❌ Parallel to it |

### My Read

**Option A is the pragmatic starting point.** It requires minimal changes and leverages everything Remora already does. The main gap — outward scaffolding — can be covered by Grail tools that write files and emit `ContentChangedEvent`s.

**Option B is the ambitious endgame.** If self-generation becomes a first-class Remora concept, generator nodes give you the explicit semantics, provenance, and lifecycle management you'd eventually want. But it's a significant architectural addition.

**Option C is a trap.** It sounds simple but doesn't integrate with Remora's reactive model and would likely be replaced by Option A or B once the limitations become apparent.

**Recommended path**: Start with Option A, add a thin declarative "template registry" config layer on top to give it structure, and evolve toward Option B only if the use cases demand explicit provenance and ordering.

## 3. Template System Design Space

Regardless of integration model, we need a way to declare templates. Here's the design space.

### 3.1 Template Registry (Declarative Config)

Templates would live in `remora.yaml` (or a separate `templates.yaml`):

```yaml
templates:
  - name: python-project
    match:
      filename: "pyproject.toml"
      # Could also be: path_glob, content_pattern, etc.
    sections:
      # Inward expansion: which parts of the file become managed nodes
      - query: "tables"          # tree-sitter query to use
        node_type: "table"       # what node type to assign
        bundle: "pyproject-agent"
    scaffold:
      # Outward scaffolding: what to generate
      - script: "scaffold/python-layout.pym"
        trigger: "on_create"     # only on first discovery, not on changes
        approval: auto           # no human review needed
      - script: "scaffold/python-tests.pym"
        trigger: "on_create"
        approval: propose        # requires human review

  - name: devenv-config
    match:
      filename: "devenv.nix"
    sections:
      - query: "nix-attrs"
        bundle: "devenv-agent"
    scaffold:
      - script: "scaffold/devenv-setup.pym"
        trigger: "on_create"
        approval: auto
```

### 3.2 File Pattern Matching

What makes a file a "template trigger"? Options:

| Matcher | Example | Complexity |
|---------|---------|-----------|
| Exact filename | `pyproject.toml` | Trivial |
| Glob pattern | `*.toml` | Low |
| Path glob | `**/docker-compose*.yml` | Low |
| Content pattern | File contains `[tool.remora]` | Medium |
| Filename + location | `pyproject.toml` at project root only | Low |
| Composite | Filename AND content pattern | Medium |

Recommendation: Start with `filename` + optional `path_glob`. Content matching is powerful but requires reading every file, which is expensive during discovery.

### 3.3 Section Extraction

For "inward expansion," we need to identify sections within a file. Options:

**Tree-sitter queries** (already have this for TOML tables):
- Pro: Precise, language-aware
- Con: Need queries per format, limited by grammar

**Line-based patterns** (regex on content):
- Pro: Universal, simple
- Con: Fragile, doesn't understand structure

**Parser-specific** (use TOML parser for .toml, INI parser for .ini):
- Pro: Perfect structural understanding
- Con: Need a parser per format, not tree-sitter

**Hybrid** (tree-sitter for discovery, format-specific parser for detailed extraction):
- Pro: Best of both worlds
- Con: More code

Recommendation: Tree-sitter queries for section boundaries (which nodes exist), Grail scripts for detailed content extraction (what's in each section). This keeps discovery fast and generation flexible.

## 4. Trigger Mechanics

When should generation happen?

### 4.1 Trigger Types

| Trigger | Meaning | Use Case |
|---------|---------|----------|
| `on_create` | File appears for first time | Initial scaffolding |
| `on_change` | File content changes | Re-sync config |
| `on_section_add` | New section added to file | Add tooling for new dependency |
| `on_section_change` | Existing section modified | Update generated code |
| `on_section_remove` | Section deleted | Clean up generated artifacts |
| `on_manual` | User explicitly triggers | "Regenerate from template" |

### 4.2 Mapping to Remora Events

These triggers map naturally to existing Remora events:

- `on_create` → `NodeDiscoveredEvent` for the template file's node
- `on_change` → `NodeChangedEvent` for the template file's node
- `on_section_add` → `NodeDiscoveredEvent` for the section node
- `on_section_change` → `NodeChangedEvent` for the section node
- `on_section_remove` → `NodeRemovedEvent` for the section node
- `on_manual` → `AgentMessageEvent` from user to the template node

This is why Option A works so well — the trigger mechanics already exist.

### 4.3 Cascade Control

When generation creates new files that themselves match templates, we need cascade control:

- **Depth limit**: Max N levels of cascading (like `max_trigger_depth`)
- **Seen set**: Don't re-trigger a template that's already been processed this cycle
- **Explicit opt-in**: Generated files must contain a marker (e.g., `# remora:template=xyz`) to trigger further generation
- **Dependency DAG**: Templates declare their dependencies, system topologically sorts

Recommendation: Use the existing `max_trigger_depth` + correlation_id tracking. A generated file gets a `ContentChangedEvent` with the same correlation_id as its generator, so depth limits naturally apply.

## 5. Generation Workflows

What does a generation script actually do?

### 5.1 Scaffold Script Anatomy

A Grail script (`.pym`) for scaffolding might look like:

```python
# scaffold/python-layout.pym
# Generate Python project layout from pyproject.toml

# Inputs from the template system
input source_content: str     # The seed file's content
input project_root: str       # Absolute path to project root

# Externals (capabilities injected by Remora)
external write_file           # Write to agent workspace
external read_file            # Read from agent workspace
external propose_changes      # Submit for review

import tomllib

config = tomllib.loads(source_content)
project_name = config.get("project", {}).get("name", "myproject")
packages = config.get("tool", {}).get("hatch", {}).get("build", {}).get("targets", {}).get("wheel", {}).get("packages", [f"src/{project_name}"])

for pkg_path in packages:
    await write_file(f"source/{pkg_path}/__init__.py", f'"""Package {project_name}."""\n')

await write_file(f"source/tests/__init__.py", "")
await write_file(f"source/tests/conftest.py", '"""Shared test fixtures."""\n')

result = f"Scaffolded {project_name}: {len(packages)} packages + tests"
result
```

### 5.2 Section Agent Script Anatomy

A Grail script for managing a config section:

```python
# tools/sync-ruff-config.pym
# Keep ruff configuration in sync with project conventions

input node_source: str   # The [tool.ruff] section content

external read_file
external write_file
external graph_query_nodes

# Check if ruff.toml exists alongside the pyproject
existing_ruff = await read_file("source/ruff.toml")

# If pyproject has [tool.ruff] AND there's a standalone ruff.toml, flag conflict
if existing_ruff and node_source:
    result = "CONFLICT: ruff config exists in both pyproject.toml and ruff.toml"
else:
    result = "Ruff config is consistent"
result
```

### 5.3 Generation Output Types

What can a generation workflow produce?

| Output | Mechanism | Example |
|--------|-----------|---------|
| New files on disk | `write_file` → `propose_changes` → accept | Scaffolded `src/` layout |
| New nodes in graph | `event_emit(NodeDiscoveredEvent)` | Register a virtual coordinator |
| New subscriptions | `event_subscribe(...)` | Watch for related changes |
| Events to other agents | `send_message(target, content)` | Tell the CI agent to update |
| KV metadata | `kv_set(key, value)` | Record generation state |

## 6. Execution Model

### 6.1 Approval Modes (Configurable Per-Template)

```yaml
templates:
  - name: python-project
    scaffold:
      - script: "scaffold/layout.pym"
        approval: auto              # Just do it
      - script: "scaffold/ci.pym"
        approval: propose           # Human reviews diff
      - script: "scaffold/deploy.pym"
        approval: confirm           # Human clicks "yes" (no diff review)
```

**`auto`**: Script runs, files are written directly. Used for low-risk structural scaffolding (empty `__init__.py`, directory creation).

**`propose`**: Script runs in the agent workspace, results go through the existing `RewriteProposal` flow. Human sees diffs and approves/rejects. Used for generated code that matters.

**`confirm`**: Script shows what it plans to do, human confirms. No diff, just intent. Used for operations like "add dependency X to pyproject.toml".

### 6.2 Idempotency

Generation scripts should be idempotent — running them twice produces the same result. This means:

- Check if output files exist before creating
- Use content hashing to skip unchanged files
- Store generation state in KV (`kv_set("last_generated_hash", ...)`)

### 6.3 Conflict Resolution

What happens when a generated file has been manually edited?

| Strategy | Behavior |
|----------|----------|
| **Overwrite** | Always regenerate (dangerous) |
| **Skip** | Don't touch manually-edited files |
| **Merge** | Three-way merge: original generated + manual edits + new generated |
| **Warn** | Emit event, let human decide |
| **Marker-based** | Only regenerate sections between `# BEGIN GENERATED` / `# END GENERATED` markers |

Recommendation: Default to **warn** (emit event, set node to `AWAITING_REVIEW`). Allow templates to override to **skip** or **marker-based** for specific outputs.

## 7. Concrete Examples

### 7.1 Example: New Python Package

**Seed file**: User creates `pyproject.toml` at project root.

**Template match**: `filename: pyproject.toml`

**Inward expansion**:
- Tree-sitter TOML parser discovers tables: `[project]`, `[build-system]`, `[tool.ruff]`, `[tool.pytest]`
- Each table becomes a node: `pyproject.toml::[project]`, `pyproject.toml::[tool.ruff]`, etc.
- Bundle rules assign:
  - `[project]` → `project-meta-agent` (manages name, version, dependencies)
  - `[tool.ruff]` → `linter-config-agent` (ensures ruff config is consistent)
  - `[tool.pytest]` → `test-config-agent` (validates pytest settings)

**Outward scaffolding**:
- `scaffold/python-layout.pym` reads `[project].name` and creates `src/<name>/__init__.py`
- `scaffold/python-tests.pym` creates `tests/conftest.py`, `tests/__init__.py`
- Generated files get reconciled → become their own nodes → trigger their own agents

**Cascading**:
- The generated `src/<name>/__init__.py` is a Python file → PythonPlugin discovers it → function/class nodes get created as code is written

### 7.2 Example: Devenv Configuration

**Seed file**: User creates `devenv.nix`.

**Template match**: `filename: devenv.nix`

**Inward expansion**:
- Nix grammar discovers attribute sets: `packages`, `languages.python`, `services.postgres`
- Each becomes a node that can reason about its section

**Outward scaffolding**:
- `scaffold/devenv-lockfile.pym` generates `devenv.lock`
- `scaffold/devenv-shell-hook.pym` creates `.envrc` for direnv integration

### 7.3 Example: API Spec → Implementation

**Seed file**: User creates `openapi.yaml`.

**Template match**: `filename: "openapi.yaml"` or `path_glob: "**/openapi.{yaml,yml,json}"`

**Inward expansion**:
- YAML parser discovers paths: `/users`, `/users/{id}`, `/posts`
- Each path becomes a node with an `api-endpoint-agent` bundle

**Outward scaffolding**:
- `scaffold/api-routes.pym` generates route stubs in `src/routes/`
- `scaffold/api-models.pym` generates Pydantic models from schemas
- `scaffold/api-tests.pym` generates test stubs

**Cascading**:
- Generated route files become Python code nodes
- Each function node gets a `code-agent` bundle
- Agent can fill in implementation based on the OpenAPI spec context

### 7.4 Example: Monorepo Package

**Seed file**: User creates `packages/auth/package.json`.

**Template match**: `path_glob: "packages/*/package.json"`

**This is interesting because**: The template match is location-aware. A `package.json` at root means something different than one inside `packages/`.

## 8. Open Questions

### 8.1 Ownership: Who "owns" generated files?

After scaffolding creates `src/app.py`, who manages it? Options:
- **Generator retains ownership**: Changes to the seed re-trigger generation (risk of overwriting edits)
- **Ownership transfers**: Generated file is a one-time seed, human/agents take over
- **Shared ownership**: Generator manages structure, agents manage content within structure

### 8.2 Template discovery: Where do templates live?

- In the project's `remora.yaml`? (per-project templates)
- In a global templates directory? (shared across projects)
- In the bundle system? (templates as a special bundle type)
- Shipped with Remora? (built-in templates for common patterns)

### 8.3 Template versioning

If a template changes (new version of `scaffold/python-layout.pym`), should existing projects be re-generated? How do you handle migrations?

### 8.4 Circular generation

What if template A generates a file that matches template B, which generates a file that matches template A? Need explicit cycle detection beyond depth limits.

### 8.5 Interaction with existing files

If the user already has a `src/` directory with code when they create `pyproject.toml`, should scaffolding:
- Skip (don't overwrite existing)
- Merge (add missing pieces)
- Warn (tell user about conflicts)
- Adapt (read existing structure and generate complementary files only)

### 8.6 How granular should sections be?

For pyproject.toml:
- One node per top-level table? (`[project]`, `[tool.ruff]`)
- One node per key? (`project.name`, `project.version`)
- One node per "semantic group"? (all dependency-related keys across tables)

The granularity determines how many agents are involved and how targeted their reasoning can be.

### 8.7 Should templates be able to modify the seed file itself?

Example: A `pyproject.toml` template detects that `[tool.ruff]` is missing and adds it. This creates a cycle (modifying the seed triggers re-reconciliation), but it's a useful pattern ("opinionated project templates that enforce conventions").

## Appendix: Integration Path Sketch (Option A + Template Registry)

If we go with "extend existing + thin config layer," here's what the implementation looks like:

### Config additions to `remora.yaml`:
```yaml
templates:
  - match: { filename: "pyproject.toml" }
    bundle: "pyproject-agent"
    bundle_rules:
      - { node_type: "table", name_pattern: "project", bundle: "project-meta-agent" }
      - { node_type: "table", name_pattern: "tool.*", bundle: "tool-config-agent" }
    scaffold:
      - { script: "bundles/generators/python-layout.pym", trigger: "on_create", approval: "auto" }
```

### Code changes:
1. **New `TemplateConfig` model** in `config.py` (~30 lines)
2. **Template matching in reconciler** — `_reconcile_file` checks if the file matches a template, applies bundle rules, runs scaffold scripts (~50 lines)
3. **Scaffold runner** — Executes Grail scripts with appropriate capabilities and handles approval mode (~80 lines)
4. **Template state tracking** — KV entries or a `templates` table to track what's been generated (~40 lines)

Total: ~200 lines of new code, no architectural changes.

This gets you 80% of the functionality. The remaining 20% (provenance tracking, lifecycle management, dependency ordering) would come later if needed by evolving toward Option B.
